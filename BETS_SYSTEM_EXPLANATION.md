# Betting System - Data Flow & Firebase Integration

## Overview
The betting system is **fully integrated with Firebase Firestore**. All bets, votes, and results are stored in the cloud database, not locally.

---

## 📊 Firebase Collections Used

### 1. **`bets` Collection**
Stores all created bets in the database.

**Document Structure:**
```
{
  id: "betId123",                    // Auto-generated by Firestore
  title: "Will it rain today?",      // User provided
  sideA: "Yes, it will rain",        // Option A
  sideB: "No, it won't rain",        // Option B
  amount: 100,                        // Coins per vote (user specified)
  creatorId: "userId123",             // Who created the bet
  creator: {
    name: "Alex"                      // Creator's name (for display)
  },
  voters: [
    { userId: "user1", side: "A" },
    { userId: "user2", side: "B" },
    { userId: "user3", side: "A" }
  ],
  ended: false,                      // Bet status (open vs closed)
  winningSide: null,                 // Set when bet ends
  endedAt: null,                     // Timestamp when bet was resolved
  createdAt: Timestamp(2024-...),    // When bet was created
  updatedAt: Timestamp(2024-...)     // Last update timestamp
}
```

### 2. **`users` Collection**
Stores user data including balance (updated when bets are made/won).

**Relevant Fields:**
```
{
  id: "userId123",
  name: "John",
  email: "john@example.com",
  balance: 1500,                     // Updated when voting on bets
  updatedAt: Timestamp(...)
}
```

## 🎲 How Bets Are Created & Saved

### Step 1: User Creates Bet
**Endpoint:** `POST /api/bets`

**Request:**
```json
{
  "title": "Will it rain?",
  "sideA": "Yes",
  "sideB": "No",
  "amount": 100
}
```

**Database Operations (Atomic Batch):**
1. ✅ **Create bet document** in `bets` collection
   - Set initial voters array with creator as first voter (voting for Side A)
   - Store all bet details (title, sides, amount, timestamps)
2. ✅ **Deduct coins from creator** in `users` collection
   - Reduce balance by bet amount
3. ✅ **Send notification** to all users
   - Broadcast "New Bet Created" notification

**Example After Creation:**
```
bets/xyz123
├── title: "Will it rain?"
├── sideA: "Yes"
├── sideB: "No"
├── amount: 100
├── creatorId: "user1"
├── voters: [{ userId: "user1", side: "A" }]
├── ended: false
└── createdAt: (timestamp)

users/user1
└── balance: 900  // Was 1000, now deducted
```

## 🗳️ How Votes Are Recorded & Saved

### Step 2: Users Vote on Bet
**Endpoint:** `POST /api/bets/vote`

**Request:**
```json
{
  "betId": "xyz123",
  "side": "B"
}
```

**Validations Before Saving:**
1. ✅ User exists and is authenticated
2. ✅ Bet exists in database
3. ✅ User hasn't already voted on this bet (checked against `voters` array)
4. ✅ User has enough coins to vote (balance >= bet.amount)

**Database Operations (Atomic Batch):**
1. ✅ **Add voter to bet** in `bets.voters` array
   - Append `{ userId: "user2", side: "B" }` to existing voters
2. ✅ **Deduct coins** from voting user in `users` collection
   - Reduce balance by bet amount

**Example After Vote:**
```
bets/xyz123
├── voters: [
│   { userId: "user1", side: "A" },
│   { userId: "user2", side: "B" }  ← New voter added
│]
└── updatedAt: (new timestamp)

users/user2
└── balance: 900  // Was 1000, now deducted
```

## 🏁 How Bet Results Are Calculated & Saved

### Step 3: Admin Ends Bet
**Endpoint:** `POST /api/bets/end`

**Admin-Only Request:**
```json
{
  "betId": "xyz123",
  "winningSide": "A"
}
```

**Calculations:**
```
Total voters: 2 (user1 for A, user2 for B)
Winning side: A
Winners: 1 (user1)
Total pot: 100 coins × 2 voters = 200 coins
Winnings per winner: 200 ÷ 1 = 200 coins
```

**Database Operations (Atomic Batch):**
1. ✅ **Mark bet as ended** in `bets` collection
   - Set `ended: true`
   - Set `winningSide: "A"`
   - Set `endedAt: (timestamp)`

2. ✅ **Distribute winnings** to winners in `users` collection
   - For each winner: `balance += winnings`
   - Example: user1 gets +200 coins → balance = 1100 (1000 - 100 initial - 100 vote = 800, then + 200 = 1000... wait let me recalculate)
   - Actually: user1 starts with 1000, votes (loses 100) = 900, then wins (gains 200) = 1100

3. ✅ **Send notifications**
   - Winners: "🎉 You Won! +200 TumbaCoins in the bet..."
   - Losers: "😔 Bet Ended. Side A won."

**Example After Bet Ends:**
```
bets/xyz123
├── ended: true ← Status changed
├── winningSide: "A" ← Result recorded
├── endedAt: (timestamp) ← Closed at this time
└── voters: [...]

users/user1
└── balance: 1100 ← Windows received +200

users/user2
└── balance: 900 ← No refund (lost the bet)
```

## 🔒 Data Safety & Atomicity

### Why Using Batch Writes?
All the changes above **happen atomically** (all-or-nothing). This means:
- **If something fails in the middle**, everything rolls back
- **Prevents corruption** (e.g., coins deducted but bet not created)

### Example Without Batch (DANGEROUS):
```
1. Create bet
2. ❌ ERROR: Network fails
3. Coins never deducted → BUG: Free bet!
```

### Example With Batch (SAFE):
```
1. Prepare: Create bet + Deduct coins
2. Commit atomically
   - ✅ Both succeed OR
   - ❌ Both fail (nothing partially changes)
```

## ✅ Current Validations

### When Creating Bet:
- ✅ User is authenticated
- ✅ All required fields present (title, sideA, sideB, amount > 0)
- ✅ User has enough balance to vote for their own bet

### When Voting:
- ✅ User is authenticated
- ✅ Bet exists in database
- ✅ User hasn't voted before (checked in voters array)
- ✅ User has enough balance to vote
- ✅ Side is valid ("A" or "B")

### When Ending Bet:
- ✅ User is admin only
- ✅ Bet exists
- ✅ Bet isn't already ended (prevents double-payout)
- ✅ Winning side is valid ("A" or "B")

## 🔗 How Frontend Connects to This Data

### Bet Display Flow:
1. User navigates to `/bets` page
2. Frontend calls `GET /api/bets`
3. API queries Firebase: `db.collection('bets').orderBy('createdAt', 'desc').get()`
4. Returns all bets with current voters
5. Component shows live vote counts

### Admin Endpoint Flow:
1. Admin clicks "End Bet" with winning side
2. Frontend calls `POST /api/bets/end` with betId + winningSide
3. API updates Firebase atomically
4. Notifications sent
5. Admin sees confirmation
